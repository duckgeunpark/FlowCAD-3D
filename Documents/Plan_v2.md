# 덕트·배관 3D 자동생성 서비스 설계 문서

## 개요

본 문서는 사용자가 최소한의 정보만 입력해도 덕트와 배관을 3D로 자동 생성할 수 있는 서비스의 기본 설계 방향을 정의한다. 이 구조는 덕트와 배관을 파라메트릭 부품 라이브러리와 연결 규칙으로 관리하고, 업로드된 표를 해석해 3D 모델을 조립하는 방식에 적합하다[cite:28][cite:43].

핵심 목표는 세 가지다. 첫째, 사용자의 입력 부담을 최소화한다. 둘째, 원형과 사각 형상에 맞는 연결 규칙을 시스템 내부에서 자동 판정한다. 셋째, 향후 물량 산출, 도면 생성, 간섭 검토, 장비 연결 같은 확장 기능을 붙일 수 있도록 데이터 구조를 표준화한다[cite:28][cite:49].

## 목표 범위

초기 버전은 덕트와 배관의 기본 경로 생성에 집중한다. 사용자가 입력한 자재 순서, 기본 규격, 길이 또는 각도, 연결 대상 정보만으로 직관, 엘보, 티, 레듀서 같은 기본 부품을 3D로 생성하는 수준을 목표로 한다[cite:43][cite:49].

초기 대상은 다음과 같다.
- 배관: 직관, 엘보 45도, 엘보 90도, 티, 레듀서.
- 덕트: 사각 직관, 사각 엘보, 사각 티, 사각 변환관, 원형 직관, 원형 엘보.
- 연결 판정: 원형은 직경 기준, 사각은 가로·세로 기준.

## 전체 구조

서비스는 크게 사용자 입력 계층, 부품 라이브러리 계층, 규칙 엔진 계층, 배치 엔진 계층, 3D 생성 계층으로 나뉜다. 파라메트릭 파트 카탈로그 방식은 동일한 부품 패밀리에서 크기만 바꿔 여러 부품을 만들 수 있으므로, 사용자가 모든 조인트와 형상을 직접 정의하지 않아도 된다[cite:43][cite:49].

전체 흐름은 아래와 같다.
1. 사용자가 엑셀 표를 업로드한다.
2. 시스템이 각 행을 자재 객체로 해석한다.
3. 자재 타입에 맞춰 내부 부품 라이브러리에서 파트 패밀리를 찾는다.
4. 해당 패밀리의 포트 정보를 기반으로 조인트를 자동 생성한다.
5. 연결 대상과 연결 위치를 해석해 자재 간 그래프를 만든다.
6. 규칙 엔진이 연결 가능 여부를 검사한다.
7. 배치 엔진이 위치, 방향, 회전값을 계산한다.
8. 3D 메시 또는 파라메트릭 형상을 생성한다.

## 사용자 입력 최소화 전략

입력 최소화의 핵심은 사용자가 자재의 세부 형상과 조인트 번호를 직접 입력하지 않도록 하는 것이다. 파라메트릭 모델은 형상 자체보다 크기 파라미터와 연결 정의가 중요하므로, 최소 입력 방식과 잘 맞는다[cite:43].

사용자가 직접 입력해야 하는 값은 다음 정도로 제한하는 것이 좋다.
- 자재 순서.
- 자재 종류.
- 규격.
- 길이 또는 각도.
- 어느 자재에 연결되는지.

반대로 시스템이 자동 처리해야 하는 정보는 다음과 같다.
- 조인트 생성.
- 포트 방향 벡터 계산.
- 회전 허용 여부 판정.
- 연결 가능 여부 검사.
- 3D 배치 좌표 계산.
- 필요 시 기본 길이 또는 반경 보정.

## 추천 업로드 표

초기 MVP는 사용자가 한 장의 시트만 업로드하도록 설계하는 것이 가장 단순하다. 이 표는 부품 정의서가 아니라 조립 지시표 역할을 하며, 실제 형상과 연결 규칙은 시스템 내부 라이브러리에서 가져온다[cite:28][cite:43].

### user_input 시트 권장 컬럼

| 컬럼명 | 설명 |
|---|---|
| seq | 자재 순번 |
| system_type | `pipe` 또는 `duct` |
| part_type | `straight`, `elbow`, `tee`, `reducer` 등 |
| spec | 재질/규격 그룹 코드 |
| size_a | 원형이면 직경, 사각이면 가로 |
| size_b | 사각일 때 세로, 원형이면 비움 |
| length | 직관 또는 변환관 길이 |
| angle | 엘보 각도 |
| connect_to_seq | 몇 번 자재에 연결되는지 |
| connect_port | 연결 대상 자재의 `start` 또는 `end` |
| note | 비고 |

### user_input 예시

| seq | system_type | part_type | spec | size_a | size_b | length | angle | connect_to_seq | connect_port | note |
|---|---|---|---|---:|---:|---:|---:|---:|---|---|
| 1 | pipe | straight | SCH40 | 100 |  | 3000 |  |  | start | 시작 배관 |
| 2 | pipe | elbow | SCH40 | 100 |  |  | 45 | 1 | end | 45도 방향 전환 |
| 3 | pipe | straight | SCH40 | 100 |  | 2000 |  | 2 | end | 연장 |
| 4 | duct | straight | GI | 500 | 300 | 1500 |  |  | start | 시작 덕트 |
| 5 | duct | elbow | GI | 500 | 300 |  | 90 | 4 | end | 90도 전환 |

이 방식이면 사용자는 조인트 번호를 모를 수 있어도 된다. 예를 들어 2번 자재가 1번 자재의 `end`에 연결된다고만 입력하면, 시스템은 1번 자재의 끝 포트와 2번 자재의 시작 포트를 자동으로 대응시킨다.

## 내부 데이터 변환 구조

업로드된 한 줄은 내부적으로 `part -> ports/joints -> connections` 구조로 변환되어야 한다. 파트 카탈로그와 파라메트릭 네트워크 생성 방식은 부품 정의와 연결 정의를 분리해 관리할 때 확장성과 재사용성이 높다[cite:43][cite:49].

### 내부 Part 객체 예시

```json
{
  "part_id": "P0001",
  "system_type": "pipe",
  "part_family": "P_STRAIGHT",
  "shape": "round",
  "params": {
    "diameter": 100,
    "length": 3000
  }
}
```

### 내부 Joint 객체 예시

```json
{
  "joint_id": "P0001_J2",
  "part_id": "P0001",
  "port_name": "end",
  "shape": "round",
  "size": {
    "diameter": 100
  },
  "direction": [1, 0, 0]
}
```

### 내부 Connection 객체 예시

```json
{
  "from_joint_id": "P0001_J2",
  "to_joint_id": "P0002_J1",
  "connect_type": "butt"
}
```

## 부품 라이브러리 설계

시스템 내부에는 모든 완제품을 개별 파일로 넣기보다, 자주 쓰는 형상 패밀리와 그 파라미터를 미리 정의한 부품 라이브러리를 둬야 한다. 파라메트릭 파트는 크기 파라미터에 따라 동적으로 형상이 바뀌므로, 패밀리 중심으로 구성하는 것이 효율적이다[cite:43][cite:49].

부품 라이브러리의 각 항목은 최소한 아래 정보를 가져야 한다.
- 패밀리 코드.
- 시스템 타입.
- 형상 타입.
- 포트 개수.
- 필수 파라미터 목록.
- 기본 제약 조건.
- 포트 정의.
- 연결 규칙 ID.

### 배관 부품 라이브러리 예시

| family_code | 부품명 | 포트 수 | 필수 파라미터 | 설명 |
|---|---|---:|---|---|
| P_STRAIGHT | 직관 | 2 | `diameter`, `length` | 배관 기본 직선 구간 |
| P_ELBOW_45 | 엘보 45 | 2 | `diameter`, `center_radius` | 45도 방향 전환 |
| P_ELBOW_90 | 엘보 90 | 2 | `diameter`, `center_radius` | 90도 방향 전환 |
| P_TEE | 티 | 3 | `main_dia`, `branch_dia` | 분기 |
| P_REDUCER_CONC | 동심 레듀서 | 2 | `dia_in`, `dia_out`, `length` | 중심 축 기준 축소 |
| P_REDUCER_ECC | 편심 레듀서 | 2 | `dia_in`, `dia_out`, `length`, `flat_side` | 상·하 기준 편심 축소 |
| P_CAP | 캡 | 1 | `diameter` | 끝마감 |
| P_FLANGE | 플랜지 | 2 | `diameter`, `thickness`, `rating` | 접속부 |
| P_VALVE_GATE | 게이트밸브 | 2 | `diameter`, `face_to_face` | 차단 장치 |
| P_VALVE_BUTTERFLY | 버터플라이밸브 | 2 | `diameter`, `face_to_face` | 대구경용 차단 장치 |

### 덕트 부품 라이브러리 예시

| family_code | 부품명 | 형상 | 포트 수 | 필수 파라미터 |
|---|---|---|---:|---|
| D_STRAIGHT_RECT | 사각 직관 | rect | 2 | `width`, `height`, `length` |
| D_ELBOW_RECT_45 | 사각 엘보 45 | rect | 2 | `width`, `height`, `centerline_radius` |
| D_ELBOW_RECT_90 | 사각 엘보 90 | rect | 2 | `width`, `height`, `throat_radius` |
| D_TEE_RECT | 사각 티 | rect | 3 | `main_w`, `main_h`, `branch_w`, `branch_h` |
| D_TRANSITION_RECT | 사각 변환관 | rect | 2 | `w1`, `h1`, `w2`, `h2`, `length` |
| D_OFFSET_RECT | 사각 옵셋 | rect | 2 | `width`, `height`, `offset`, `length` |
| D_STRAIGHT_ROUND | 원형 직관 | round | 2 | `diameter`, `length` |
| D_ELBOW_ROUND_90 | 원형 엘보 90 | round | 2 | `diameter`, `radius` |
| D_TRANSITION_R2R | 원형 변환관 | round | 2 | `d1`, `d2`, `length` |
| D_TRANSITION_R2RECT | 원사각 변환관 | mixed | 2 | `diameter`, `width`, `height`, `length` |

## 포트 정의 방식

실제 자동 연결에서 가장 중요한 정보는 형상보다 포트다. 포트는 어느 위치에서 어떤 단면과 방향으로 연결되는지 정의하며, 연결 가능 판정의 기준점이 된다.

### 배관 직관 포트 예시

```json
{
  "family_code": "P_STRAIGHT",
  "ports": [
    {
      "name": "start",
      "shape": "round",
      "size_keys": ["diameter"],
      "local_pos": [0, 0, 0],
      "dir": [-1, 0, 0]
    },
    {
      "name": "end",
      "shape": "round",
      "size_keys": ["diameter"],
      "local_pos": ["length", 0, 0],
      "dir": [1, 0, 0]
    }
  ]
}
```

### 사각 덕트 엘보 포트 예시

```json
{
  "family_code": "D_ELBOW_RECT_90",
  "ports": [
    {
      "name": "in",
      "shape": "rect",
      "size_keys": ["width", "height"],
      "up_ref": [0, 0, 1]
    },
    {
      "name": "out",
      "shape": "rect",
      "size_keys": ["width", "height"],
      "up_ref": [0, 0, 1]
    }
  ]
}
```

사각 부품은 단면 크기뿐 아니라 기준 상면 방향까지 유지해야 회전 정합이 쉬워진다. 원형 부품은 축 회전 자유도가 크기 때문에 직경 일치만으로 연결 판정이 가능하지만, 사각 부품은 가로·세로와 방향 정보가 함께 맞아야 한다.

## 연결 규칙

연결 규칙은 사용자 업로드 표가 아니라 시스템 내부 룰 엔진에 고정하는 편이 좋다. 이 방식은 입력 부담을 줄이고, 판정의 일관성을 유지한다[cite:28].

기본 규칙은 다음과 같다.
- 원형: 직경이 같으면 연결 가능.
- 원형: 축 회전은 자유로 본다.
- 사각: 가로와 세로가 모두 같아야 연결 가능.
- 사각: 필요 시 뒤집힘 허용 여부를 옵션으로 둔다.
- 변환관: 양쪽 포트 크기가 달라도 패밀리 정의에 맞으면 연결 가능.
- 티: 메인 포트와 브랜치 포트를 구분한다.

### 규칙 테이블 예시

| rule_id | shape | match 조건 | 회전 규칙 | 비고 |
|---|---|---|---|---|
| ROUND_SAME_DIA | round | `diameter` 동일 | 자유 | 배관, 원형 덕트 기본 |
| RECT_SAME_WH | rect | `width`, `height` 동일 | 프로파일 정렬 | 사각 덕트 기본 |
| RECT_FLIP_OPTIONAL | rect | `width`, `height` 동일 | 뒤집힘 옵션 | 선택 기능 |
| REDUCER_TRANSITION | mixed | 패밀리 정의 양쪽 포트 일치 | 포트별 검증 | 변환관용 |

## 조인트 자동 생성 규칙

사용자가 조인트 번호를 입력하지 않도록 하려면, 시스템이 각 패밀리의 포트 개수를 기준으로 조인트를 자동 생성해야 한다. 예를 들어 직관은 2포트, 티는 3포트, 캡은 1포트다.

예시 규칙은 아래와 같다.
- `straight`: `start`, `end` 자동 생성.
- `elbow`: `in`, `out` 자동 생성.
- `tee`: `main_in`, `main_out`, `branch` 자동 생성.
- `reducer`: `large_end`, `small_end` 자동 생성.
- `cap`: `in` 자동 생성.

예를 들어 사용자가 다음처럼 입력했다고 가정한다.
- 1번 자재: 직관.
- 2번 자재: 엘보 45도.
- `2번.connect_to_seq = 1`, `2번.connect_port = end`.

이 경우 시스템은 다음처럼 해석한다.
- 1번 직관에서 `end` 포트를 찾는다.
- 2번 엘보의 기본 시작 포트를 찾는다.
- 두 포트의 형상과 규격을 검사한다.
- 조건이 맞으면 연결 객체를 생성한다.

## 추천 MVP 라이브러리 범위

초기 버전에서는 모든 장비 부속을 넣기보다, 경로를 생성하는 데 꼭 필요한 부품만 우선 지원하는 것이 현실적이다. 파라메트릭 패밀리 기반 구조에서는 적은 수의 패밀리로도 많은 규격을 처리할 수 있다[cite:43].

### MVP 필수 부품

| 구분 | 필수 부품 |
|---|---|
| 배관 | 직관, 엘보45, 엘보90, 티, 레듀서 |
| 덕트 사각 | 직관, 엘보90, 티, 변환관 |
| 덕트 원형 | 직관, 엘보90, 변환관 |

### 2차 확장 부품

| 구분 | 확장 부품 |
|---|---|
| 배관 | 유니온, 커플링, 플랜지, 스트레이너, 체크밸브, 펌프 접속부 |
| 덕트 | 댐퍼, 플렉시블 조인트, 캡, 디퓨저 접속부, 분기관 |
| 공통 | 행거, 서포트, 장비 노즐, 점검구 |

## 시스템 내부 저장 구조 제안

구현 관점에서는 최소한 세 종류의 저장소가 필요하다.
- `part_families`: 부품 패밀리 정의.
- `family_sizes`: 규격별 기본값 정의.
- `projects/instances`: 업로드 후 생성된 실제 프로젝트 객체.

### part_families 예시

| family_code | system_type | family_name | shape_type | port_count | param_schema |
|---|---|---|---|---:|---|
| P_STRAIGHT | pipe | 직관 | round | 2 | `diameter,length` |
| P_ELBOW_45 | pipe | 엘보45 | round | 2 | `diameter,center_radius` |
| P_TEE | pipe | 티 | round | 3 | `main_dia,branch_dia` |
| D_STRAIGHT_RECT | duct | 사각직관 | rect | 2 | `width,height,length` |
| D_TRANSITION_R2RECT | duct | 원사각변환 | mixed | 2 | `diameter,width,height,length` |

### family_sizes 예시

## 구현 단계 제안

서비스는 한 번에 완성형으로 만들기보다 단계적으로 구현하는 것이 안정적이다.

### 1단계: 배관 MVP
- 원형 직관, 엘보, 티, 레듀서 구현.
- 직경 일치 규칙 구현.
- 업로드 표 1시트 파서 구현.
- 포트 자동 생성 및 연결 그래프 구현.
- 기본 3D 미리보기 구현.

### 2단계: 사각 덕트 확장
- 사각 직관, 엘보, 티, 변환관 구현.
- 가로·세로 일치 규칙 구현.
- 회전 정합 처리 구현.
- 원형 덕트와 사각 덕트 분리 지원.

### 3단계: 사용성 강화
- 업로드 템플릿 자동 다운로드.
- 입력 오류 행 하이라이트.
- 연결 불가 사유 표시.
- 자동 치수 보정 또는 추천값 표시.
- 결과물 저장 및 재편집 기능.

### 4단계: 고도화
- 밸브, 댐퍼, 플렉시블 조인트 추가.
- 물량 산출 BOM 생성.
- 장비 포트 연결.
- IFC/BIM 데이터 연계 검토.

## 검증 포인트

초기 구현에서는 다음 항목을 우선 검증해야 한다.
- 같은 규격의 원형 직관과 엘보가 올바르게 연결되는지.
- 사각 덕트에서 가로·세로가 다르면 연결이 차단되는지.
- 티와 레듀서처럼 포트 의미가 다른 부품이 올바르게 처리되는지.
- 업로드 표에서 누락 컬럼, 빈 값, 잘못된 규격이 명확히 검출되는지.
- 사용자가 조인트를 몰라도 자재 순서만으로 기본 경로가 생성되는지.

## 권장 결론

이 서비스는 사용자에게 복잡한 CAD 수준의 데이터를 요구하기보다, 시스템 내부에 파라메트릭 부품 라이브러리와 연결 규칙을 충분히 준비해 두고, 사용자는 조립 순서와 핵심 치수만 입력하는 구조로 설계하는 것이 가장 적절하다[cite:28][cite:43][cite:49].

초기 업로드 포맷은 `user_input` 1시트로 시작하고, 내부적으로는 `parts`, `ports`, `connections` 구조로 변환하는 방식이 가장 구현 난이도와 확장성의 균형이 좋다. 부품 라이브러리는 먼저 배관과 덕트의 기본 패밀리만 구축한 뒤, 2차 단계에서 밸브, 댐퍼, 장비 접속류를 늘려가는 전략이 현실적이다[cite:43][cite:42][cite:44].
